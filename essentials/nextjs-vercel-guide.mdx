## What Next.js is

Next.js is a React framework that supports server-side rendering, static generation, routing, and performance optimizations out of the box.

## What Vercel is

Vercel is a cloud platform designed for front-end frameworks and static sites. It provides automated builds, preview deployments, and global CDN delivery.

# Prerequisites

Ensure the following before starting:

- Node.js installed (LTS recommended)
- Git installed and configured
- A GitHub account
- A Vercel account

> **Tip:** Use the latest LTS version of Node.js for compatibility with Next.js.

# Creating a Next.js Application

Follow these steps to create and run a new Next.js application.

1. Scaffold a new Next.js app:

```bash
npx create-next-app@latest my-app
```

2. Change into the project directory:

```bash
cd my-app
```

3. Start the development server:

```bash
npm run dev
```

4. Open your browser at `http://localhost:3000` to view the running app.

## Brief project structure (summary)

- `pages/` — route files; each file maps to a route
- `public/` — static assets
- `styles/` — CSS modules and global styles
- `next.config.js` — optional Next.js configuration
- `package.json` — scripts and dependencies

> **Note:** `npm run dev` runs a development server. For production, build with `npm run build` and run the production server with `npm start` (or let Vercel handle builds).

# Setting Up a GitHub Repository

Create a repository on GitHub and push your local project.

1. On GitHub: create a new repository (for example, `my-app`).

2. From your project directory, run:

```bash
git init
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin <repo-url>
git push -u origin main
```

Replace `<repo-url>` with your repository URL, e.g. `https://github.com/your-username/my-app.git`.

📸 Screenshot: GitHub new repository creation page (repository name, visibility, create button).

📸 Screenshot: Terminal showing `git push -u origin main` completing successfully.

> **Tip:** If the remote repository already has commits (e.g., a README), run `git pull origin main --allow-unrelated-histories` before pushing.

# Connecting the Project to Vercel

Use Vercel to import the GitHub repository and enable continuous deployments.

1. Log in to Vercel at https://vercel.com (use GitHub OAuth for straightforward integration).

2. Click **New Project** → **Import Git Repository** and select your repository.

3. Vercel should automatically detect Next.js. Confirm or adjust build settings:

- Framework Preset: **Next.js**
- Build Command: `npm run build` (or `next build`)
- Output Directory: `.next`

4. Configure any required Environment Variables (see next section) and click **Deploy**.

📸 Screenshot: Vercel import repository screen showing framework detected as Next.js.

📸 Screenshot: Vercel build settings modal before deploying.

> **Note:** Vercel creates Preview Deployments for branches and PRs automatically.

# Managing Environment Variables

What environment variables are:

- Environment variables store configuration and secrets (API keys, database URLs) outside source control.

## Local setup

- Use `.env.local` for local development-only variables.
- Client-exposed variables must be prefixed with `NEXT_PUBLIC_`.

### Example `.env.local`

```bash
# Server-only (do not expose)
DATABASE_URL="postgres://user:password@host:5432/dbname"

# Client-exposed
NEXT_PUBLIC_API_BASE_URL="https://api.example.com"
```

## Adding variables in Vercel dashboard

1. Open your project in Vercel → **Settings** → **Environment Variables**.
2. Click **Add** and enter the **Name**, **Value**, and **Environment** (Preview / Production / Development).
3. Save and then redeploy the project.

## Redeploying after changes

- After adding or updating variables in Vercel, trigger a redeploy to apply changes.
- For local edits to `.env.local`, restart the dev server (`npm run dev`).

> **Tip:** Use distinct values for Preview and Production environments to avoid leaking test credentials.

# Deployment Process Explained

## Automatic deployments

- Vercel builds and deploys on each push to connected branches.
- Each PR/branch receives a Preview Deployment with an immutable URL.
- Production deployment occurs when you merge/push to the production branch (commonly `main`).

## Production builds

- Vercel runs the configured build command (e.g., `npm run build`) and serves optimized assets via a global CDN.

## Preview deployments

- Useful for QA and stakeholder review; each has an isolated URL.

## Getting the live URL

- After deployment completes, Vercel displays the deployment URL and the production URL (if applicable).

> **Note:** Configure custom domains in Vercel under the project **Domains** settings.

# Common Issues & Troubleshooting

## Issue 1 — Build failure

- Symptom: Build fails during `npm run build` on Vercel.
- Possible causes:
  - Missing dependency in `package.json`
  - Syntax error or incompatible Node.js version
  - Missing environment variable required at build time
- Resolution steps:
  1. Inspect Vercel build logs for the error message.
  2. Add missing dependencies and commit.
  3. Set the Node version in `package.json` (`engines.node`) or Vercel settings.
  4. Add any required build-time env vars in Vercel and redeploy.

## Issue 2 — Environment variable not available in browser

- Symptom: Client code cannot access a variable.
- Cause: Variable is not prefixed with `NEXT_PUBLIC_`.
- Resolution:
  - Prefix the variable with `NEXT_PUBLIC_` if it is safe for client exposure.
  - Restart the dev server and redeploy.

## Issue 3 — GitHub repository not connected in Vercel

- Symptom: Vercel cannot find or access your repository.
- Possible causes:
  - GitHub integration lacks permissions for the repository
  - The Vercel GitHub app needs reauthorization
- Resolution:
  1. Re-authorize GitHub in Vercel: Profile → Account Settings → Git Integrations.
  2. Grant the Vercel GitHub app access to the repository.

> **Tip:** For intermittent issues, check GitHub and Vercel status pages.

> **Warning:** Never commit `.env.local` or other secret values to version control. Add `.env.local` to `.gitignore`.
